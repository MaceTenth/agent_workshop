What gets painful at scale?
Writing your first LLM call in Python is 5 lines. But the moment you build something real — with memory, tools, retries, multiple agents, and structured output — the boilerplate explodes. Frameworks like LangChain and CrewAI exist to absorb that complexity so you can focus on the logic that matters.
📝
Prompt Management
You end up with f-strings scattered everywhere. Changing a system prompt means hunting through ten files. No versioning, no reuse, no easy injection of variables.
Raw Python: f-strings, manual concatenation Framework: PromptTemplate, hub.pull()
🔗
Chaining Calls
Multi-step pipelines (summarise → translate → format) require manually threading output from one call into the next. Error handling compounds with every new step.
Raw Python: nested variables, manual wiring Framework: chain = prompt | llm | parser
🧠
Memory Across Sessions
Keeping conversation history is easy for one turn, but persisting it to a database, summarising old turns to stay within context limits, and injecting it correctly takes real engineering work.
Raw Python: lists + custom DB logic Framework: ConversationBufferMemory, Redis integration
🔧
Tool / Function Calling
Defining tools, writing JSON schemas, running the agentic loop, catching errors, and feeding results back is 80+ lines of boilerplate — written fresh for every new project.
Raw Python: manual schema + while-loop Framework: @tool decorator + AgentExecutor
🏗️
Structured Output
Getting the model to always return valid JSON with the exact fields you need requires prompt tricks, manual parsing, and retry logic when the model hallucinates extra keys.
Raw Python: json.loads() + try/except Framework: .with_structured_output(MyModel)
🤝
Multi-Agent Coordination
When you need a Researcher agent to hand off to a Writer agent to hand off to a Reviewer agent, you're suddenly building an orchestration system. Task queues, context passing, and role enforcement from scratch.
Raw Python: 200+ lines of orchestration code Framework: Crew([researcher, writer, reviewer])
LangChain — composable building blocks
LangChain wraps LLM providers into a unified interface and lets you compose them with LCEL (LangChain Expression Language) — a pipe-based syntax where every component is chainable. Click each comparison to see what changes.
😤 Raw Python
def ask_llm(topic, level):
    system = f"You are an expert in {topic}."
    user   = f"Explain {topic} assuming a {level} audience."

    # Duplicated every time. No reuse. Easy to break.
    response = client.messages.create(
        model="claude-sonnet-5",
        max_tokens=1024,
        system=system,
        messages=[
            {"role": "user", "content": user},
        ]
    )
    return response.content[0].text

# Every call site must know the template structure
result = ask_llm("machine learning", "beginner")
✅ With LangChain
from langchain_anthropic import ChatAnthropic
from langchain_core.prompts import ChatPromptTemplate

# Defined once, reused everywhere, versioned in Hub
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are an expert in {topic}."),
    ("human",  "Explain {topic} assuming a {level} audience."),
])

llm   = ChatAnthropic(model="claude-sonnet-5")
chain = prompt | llm   # LCEL pipe operator

# Clean call site — just pass the variables
result = chain.invoke({"topic": "machine learning", "level": "beginner"})

✅ With LangChain (LCEL)
from langchain_anthropic import ChatAnthropic
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

llm    = ChatAnthropic(model="claude-sonnet-5")
parser = StrOutputParser()

summarise_chain = (
    ChatPromptTemplate.from_template("Summarise: {text}")
    | llm | parser
)
translate_chain = (
    ChatPromptTemplate.from_template("Translate to French: {text}")
    | llm | parser
)

# Composable pipeline — swap, extend, or branch any step
pipeline = summarise_chain | (
    lambda s: {"text": s}
) | translate_chain

result = pipeline.invoke({"text": long_article})

✅ With LangChain
from langchain_core.tools import tool
from langchain_anthropic import ChatAnthropic
from langchain.agents import AgentExecutor, create_tool_calling_agent
from langchain_core.prompts import ChatPromptTemplate

# Schema auto-generated from docstring + type hints
@tool
def get_datetime() -> str:
    "Returns current date and time."
    from datetime import datetime
    return datetime.now().isoformat()

llm   = ChatAnthropic(model="claude-sonnet-5")
agent = create_tool_calling_agent(llm, [get_datetime], prompt)

# Loop, tool dispatch, error handling — all handled
executor = AgentExecutor(agent=agent, tools=[get_datetime])
result   = executor.invoke({"input": "What time is it?"})

CrewAI — role-based multi-agent teams
CrewAI models the way human teams work: each Agent has a role, a goal, and a backstory. Tasks are assigned to specific agents, and a Crew orchestrates execution — sequentially or in parallel — without you writing any coordination code.
🤖 Agent
role: "Senior Researcher"
goal: "Find accurate info about {topic}"
backstory: "Expert at web research..."
tools: [SerperTool(), WebsiteTool()]
+
📋 Task
description: "Research {topic} and summarise findings"
agent: researcher
expected_output: "3-paragraph summary"
→
🚀 Crew
agents: [researcher, writer, reviewer]
tasks: [research_task, write_task, review_task]
process: Process.sequential
😤 Raw Python
# 3 agents, each with their own loop, manually passing context
def run_researcher(topic):
    return simple_llm_call(
        f"You are a researcher. Research: {topic}")

def run_writer(research_result):
    return simple_llm_call(
        f"You are a writer. Write an article based on:\n{research_result}")

def run_reviewer(article):
    return simple_llm_call(
        f"You are a reviewer. Review and improve:\n{article}")

# Manual orchestration — context passing, no role enforcement
research = run_researcher("quantum computing")
article  = run_writer(research)
final    = run_reviewer(article)
# No visibility, no retries, no tool access, no delegation
✅ With CrewAI
from crewai import Agent, Task, Crew, Process
from crewai_tools import SerperDevTool

researcher = Agent(
    role="Senior Researcher",
    goal="Find accurate facts about {topic}",
    backstory="Expert at sifting through web sources",
    tools=[SerperDevTool()],  # real web search built in
)
writer = Agent(role="Tech Writer", goal="Craft clear articles", ...)
reviewer = Agent(role="Editor", goal="Ensure accuracy and clarity", ...)

research_task = Task(
    description="Research {topic} and compile key findings",
    expected_output="Bullet-point summary of 5 key facts",
    agent=researcher,
)
# Tasks automatically pass output to the next task
crew = Crew(
    agents=[researcher, writer, reviewer],
    tasks=[research_task, write_task, review_task],
    process=Process.sequential,
    verbose=True,  # logs every step
)
result = crew.kickoff(inputs={"topic": "quantum computing"})

LangGraph — agents as state graphs
LangGraph models an agent as a graph: nodes do the work, edges decide what runs next, and a shared state flows through. You get loops, branching, persistence, and human-in-the-loop for free — the control flow you'd otherwise hand-roll with while loops and if statements.
😤 Raw Python
# Hand-roll the loop: call, check for tools, run, feed back, repeat
messages = [{"role": "user", "content": task}]
while True:
    resp = client.messages.create(model=MODEL, messages=messages, tools=TOOLS)
    if resp.stop_reason != "tool_use":
        break
    messages.append({"role": "assistant", "content": resp.content})
    results = [run_tool(b) for b in resp.content if b.type == "tool_use"]
    messages.append({"role": "user", "content": results})
# No persistence, no branching, no pause & resume
✅ With LangGraph
from langgraph.graph import StateGraph, START, END
from langgraph.prebuilt import ToolNode, tools_condition
from langgraph.checkpoint.memory import MemorySaver
from langchain_anthropic import ChatAnthropic

llm = ChatAnthropic(model="claude-sonnet-5").bind_tools(tools)

def call_model(state):
    return {"messages": [llm.invoke(state["messages"])]}

graph = StateGraph(State)
graph.add_node("model", call_model)
graph.add_node("tools", ToolNode(tools))
graph.add_edge(START, "model")
graph.add_conditional_edges("model", tools_condition)  # → tools or END
graph.add_edge("tools", "model")                        # loop back
app = graph.compile(checkpointer=MemorySaver())         # persistence, free
result = app.invoke({"messages": [("user", task)]})
😤 Raw Python
# Branching logic tangled into if/elif inside your loop
intent = classify(user_msg)
if "refund" in intent:
    handle_refund(user_msg)      # a separate code path
elif "status" in intent:
    check_status(user_msg)
else:
    fallback(user_msg)
# Adding a branch means editing the loop — no shared
# state, no diagram, no resume-from-here
✅ With LangGraph
def route(state) -> str:
    intent = state["intent"]
    return intent if intent in ("refund", "status") else "fallback"

graph.add_conditional_edges(
    "classify",
    route,                        # returns the next node's name
    {"refund": "refund_node",
     "status": "status_node",
     "fallback": "fallback_node"},
)
# Each branch is a node. The graph is declarative, can be
# drawn — app.get_graph().draw_mermaid() — and resumed mid-run.
Pydantic AI — type-safe agents
Pydantic AI brings the Pydantic validation you already use to agents: declare a typed output and the response is parsed and validated into a real object, register tools as plain typed functions, and inject typed dependencies. It's model-agnostic — the same agent runs on Claude.
😤 Raw Python
import json
# Ask for JSON, then hope, strip, parse, and validate by hand
prompt = f"Extract name and age from: {text}\nReturn JSON only."
raw = simple_llm_call(prompt)
raw = raw.strip("`").lstrip("json").strip()   # strip markdown fences
try:
    data = json.loads(raw)                     # might throw
    name, age = data["name"], int(data["age"])  # KeyError? ValueError?
except Exception:
    ...  # now what?
✅ With Pydantic AI
from pydantic import BaseModel
from pydantic_ai import Agent

class Person(BaseModel):
    name: str
    age:  int

agent = Agent("anthropic:claude-sonnet-5", output_type=Person)

result = agent.run_sync(f"Extract the person from: {text}")
person = result.output          # ← a validated Person, guaranteed
print(person.name, person.age)  # typed attributes, zero parsing

from pydantic_ai import Agent, RunContext

agent = Agent("anthropic:claude-sonnet-5", deps_type=WeatherClient)

@agent.tool
def get_weather(ctx: RunContext[WeatherClient], city: str) -> str:

LlamaIndex — RAG over your data
LlamaIndex is the data framework for LLM apps: point it at your documents and it handles chunking, embedding, indexing, and retrieval, then answers questions grounded in your data. What the workshop's RAG module does by hand — retrieve → augment → generate — LlamaIndex does in a few lines, with real vector search.
😤 Raw Python
# You wire up every step of RAG yourself
docs   = load_files("./data")                      # read + parse
chunks = [c for d in docs for c in split(d, 500)]  # chunk
store  = FaissIndex([embed(c) for c in chunks])       # embed + store

def answer(question):
    hits    = store.search(embed(question), top_k=3)   # retrieve
    context = "\n".join(chunks[i] for i in hits)     # stuff
    return simple_llm_call(f"Context:\n{context}\n\nQ: {question}")
# You own chunking, embeddings, storage, retrieval, and the prompt
✅ With LlamaIndex
from llama_index.core import VectorStoreIndex, SimpleDirectoryReader, Settings
from llama_index.llms.anthropic import Anthropic

Settings.llm = Anthropic(model="claude-sonnet-5")

docs  = SimpleDirectoryReader("./data").load_data()   # read + parse
index = VectorStoreIndex.from_documents(docs)          # chunk + embed + store
query = index.as_query_engine(similarity_top_k=3)      # retrieval + prompt built in

answer = query.query("What were Q3 revenues?")
# Chunking, embeddings, the vector store, retrieval, and the
# grounded prompt are all handled — swap the store or top_k with one arg

When frameworks help — and when they don't
Frameworks aren't always the right answer. Here's an honest comparison to help you decide when to reach for a framework versus writing leaner custom code.
⚙️ Raw Python (our workshop approach)
Strengths
  • Zero abstraction — you see every token and every request
  • No dependency surprises or breaking changes
  • Minimal overhead — faster cold starts
  • Full control over prompts and retry logic
  • Ideal for learning and prototyping single features
Limitations
  • Boilerplate multiplies fast as features are added
  • No built-in integrations (vector stores, search, etc.)
  • Memory, retries, streaming all require custom code
  • Multi-agent needs significant orchestration work
🦜 LangChain
Strengths
  • 150+ integrations: vector DBs, search, databases
  • LCEL makes pipelines composable and readable
  • LangSmith tracing for observability out of the box
  • Huge ecosystem: retrieval, structured output, streaming
  • Active community — almost every problem is solved
Limitations
  • Can be over-engineered for simple tasks
  • Abstractions sometimes hide important details
  • Breaking changes between major versions historically
  • Not ideal for strict latency requirements
🤝 CrewAI
Strengths
  • Purpose-built for multi-agent workflows
  • Sequential and hierarchical process modes
  • Role/goal/backstory prompting baked in
  • Built-in tools: search, scraping, file I/O
  • Human-in-the-loop support
Limitations
  • More opinionated — harder to customise deeply
  • Younger project — fewer production case studies
  • Overhead per-agent can be expensive in tokens
  • Overkill for single-agent use cases
